# 測試

Aiken 對單元測試和基於屬性的測試 (property-based tests) 提供了一等公民支援。這意味著您可以直接在 Aiken 中撰寫測試並隨時執行。因此，工具集 (`aiken check`) 可以解析測試、收集測試、執行測試並顯示包含（希望）有用的詳細資訊報告。

## 單元測試

### 撰寫單元測試

要撰寫單元測試，請使用 `test` 關鍵字：

```aiken
test foo() {
  1 + 1 == 2
}
```

單元測試是一個不接收參數並返回布林值的具名函式。更具體地說，如果測試返回 `True`，則被視為「有效」（即通過）。

您可以在 Aiken 模組的任何位置撰寫測試，它們可以以同樣的方式呼叫函式和使用常數。測試的一個令人興奮之處在於，它們使用與鏈上執行合約相同的虛擬機。換句話說，它們是您可以在與生產環境程式碼相同的上下文中執行和推理的鏈上程式碼片段。

### 單元測試報告

讓我們撰寫一個帶有一些單元測試的簡單函式作為例子。

```aiken filename="lib/example.ak"
fn add_one(n: Int) -> Int {
  n + 1
}

test add_one_1() {
  add_one(0) == 1
}

test add_one_2() {
  add_one(-42) == -41
}
```

在我們的項目中執行 `aiken check` 會得到以下報告：

![](/language_tour_tests_fig_1.png)

如您所見，報告按模組對測試進行分組，並為您提供了每個測試所需的記憶體和 CPU 執行單元。這意味著如果您需要嘗試不同的方法並比較它們的執行成本，測試也可以作為基準測試 (benchmarks)。

<Callout type="info">
  測試可以任意複雜；與鏈上腳本不同，它們沒有任何執行限制 —— 或者更準確地說，它們的限制足夠大。
</Callout>

### 自動過濾差異 (Automatic diffing)

Aiken 的測試執行器（試圖）保持智能且提供幫助，特別是在測試失敗時。如果測試失敗，執行器將盡力提供有關出錯原因的資訊。如果您使用二元運算子（`==`、`>=`、`!=` 等）將測試撰寫為斷言，這將特別有效。

例如，讓我們在上述範例中添加一個失敗的測試：

```aiken filename="lib/example.ak"
// ... 檔案其餘部分保持不變
test add_one_3() {
  add_one(1) == 1
}
```

![](/language_tour_tests_fig_2.png)

太棒了！我們可以看到兩個運算元的估值結果，測試執行器指出了問題所在。

## 基於屬性的測試 (Property-based test)

### 簡報

Aiken 的獨特優點之一是其整合了縮減 (shrinking) 功能的基於屬性的測試框架。基於屬性的測試是透過探索可能輸入的領域並尋找一般行為而非特定案例來生成測試用例的藝術。

例如，如果您考慮一個反轉列表元素順序的函式 `list.reverse`，它具有一個優良的屬性：呼叫它兩次會使列表恢復到其原始順序。您可以透過生成隨機列表樣本並檢查它們是否滿足該屬性來測試該函式。

如果發現該屬性的反例，框架會嘗試將其簡化為最小反例，以便更容易消化和推理。這是一個關鍵步驟，因為隨機探索輸入域可能會導致任意大的樣本值，從而掩蓋真正的問題。想像一個處理整數列表並在遇到負整數值時失敗的函式。在此範例中，列表 `[12, 441, 0, 7863, -2, 1213]` 是一個有效的反例，但 `[-1]` 顯然是一個更好的反例。它能讓您更直接地精確鎖定問題。

在傳統的基於屬性的測試框架中，尋找較小的反例可能是繁琐的，這被稱為「縮減 (shrinking)」。撰寫屬性的開發者必須定義如何從初始值獲得較小的反例。在 Aiken 中，框架整合並自動管理這一過程。

### 撰寫屬性

基於屬性的測試是帶有一個指定 `Fuzzer` 參數的測試。模糊測試器 (fuzzer) 或生成器是一個抽象，它指定了如何從隨機源生成（偽）隨機值。

我們提供 —— *並強烈建議使用* —— 一個核心模糊測試器庫：[`aiken/fuzz`](https://github.com/aiken-lang/fuzz)。該庫包含了寶貴的原語，代表您處理抽象偽隨機性管理的繁琐工作，使撰寫模糊測試器變得輕鬆愉快。請盡情使用！

```aiken
use aiken/fuzz

test prop_is_non_negative(n: Int via fuzz.int()) {
  n >= 0
}
```

`Fuzzer` 作為參數的一個特殊標註，使用 `via` 關鍵字引入，且必須是型別為 `Fuzzer<a>`（儘管 `a` 必須實例化為具體型別！）。在上述範例中，它的型別是 `Fuzzer<Int>`。因此，參數的型別與其模糊測試器所攜帶的型別之間存在直接關係！因此，參數的型別標註是完全選填的，因為它與模糊測試器是冗餘的。

#### 組合模糊測試器

模糊測試器可以在 via 關鍵字之後直接行內組合：

```aiken
use aiken/fuzz

test prop_list(xs: List<Int> via fuzz.list(fuzz.int())) {
  todo
}
```

它們也可以定義在單獨的函式中並透過名稱呼叫：

```aiken
use aiken/fuzz

fn my_fuzzer() -> Fuzzer<List<Int>> {
  fuzz.list(fuzz.int())
}

test prop_list(xs: List<Int> via my_fuzzer()) {
  todo
}
```

### 屬性測試報告

與單元測試類似，屬性測試使用 `aiken check` 命令執行。它們提供一份報告，強調檢查的隨機樣本數量，並在失敗情況下提供簡化的反例。例如，如果我們對上述處理整數的 `prop_is_non_negative` 執行測試，我們會得到以下結果：

![](/language_tour_tests_fig_3.png)

如您所見，該屬性被 `int()` 模糊測試器生成的負值判定為無效。反例直接顯示在測試報告中，以及在找到反例之前執行的測試次數。

### 標籤 (Labelling)

通常，人們想評估隨機遊走是否探索了特定的路徑。對於探索大型定義域空間的複雜生成器來說尤其如此。[`aiken/fuzz`](https://github.com/aiken-lang/fuzz) 庫為此提供了「標籤 (labels)」，框架會自動收集這些標籤。測試成功後，它們在所有測試執行中的分佈將顯示在報告中。標籤對於評估模糊測試器的正確性也非常有用，我們強烈建議**測試您的模糊測試器**！基於屬性測試的一個常見陷阱是，由於模糊測試器定義錯誤，屬性實際上沒有測試任何內容。例如，讓我們檢查 [`aiken/fuzz`](https://github.com/aiken-lang/fuzz) 庫中的 `bool()` 模糊測試器：

```aiken
test prop_bool_distribution(predicate via fuzz.bool()) {
  fuzz.label(
    if predicate {
      @"True"
    } else {
      @"False"
    },
  )

  True
}
```

我們可以看到，生成器在 `True` 和 `False` 之間基本是均勻分佈的（呼！）。請注意，由於樣本的隨機性，分佈不一定正好是 50%，但它會漸進地收斂於此。

![](/language_tour_tests_fig_4.png)

<Callout type="info">
  請注意，我們在這裡向 `check` 命令傳遞了額外的選項 `--max-success=1000`，以增加測試執行的次數並更好地觀察隨機樣本。
</Callout>

## 測試失敗情況

有時，您需要斷言特定的執行路徑「可以失敗」。這也被稱為「預期的失敗 (expected failure)」，是斷言程式行為的一種有效方式。幸運的是，您也可以在 Aiken 中透過在測試名稱後添加 `fail` 來實現此目的。例如：

```aiken filename="lib/example.ak"
use aiken/math

test must_fail() fail {
  expect Some(result) = math.sqrt(-42)
  result == -1
}
```

這裡的 `fail` 關鍵字對單元測試和屬性測試都有效，但屬性測試有一點細微的區別。事實上，預期失敗的屬性測試仍會執行多次，且只有當屬性的「所有執行」都失敗時，才被視為成功。這對於撰寫非屬性 (non-properties) 很有用，特別是如果它們在底層依賴於 `fail` 或 `expect` 來檢查前提條件。換句話說，標記為 `fail` 的屬性測試將在第一次成功求值時停止並被標記為失敗。

對於屬性測試，還可以指定 `fail once`，以便執行屬性直到它失敗一次或達到最大嘗試次數（預設為 100）。如果執行器找到了 100 個滿足預期「至少失敗一次 (fail once)」的屬性的測試，則整個測試被視為失敗。換句話說，標記為 `fail once` 的屬性測試將在第一次失敗求值時停止並被標記為成功。如果沒有評估失敗，它將被標記為失敗。

## 執行特定測試

`aiken check` 支援旗標，允許您執行專案中所有測試的子集。

### 範例

`aiken check -m "aiken/list"`

這僅執行名為 `aiken/list` 的模組內的測試。

---

`aiken check -m "aiken/option.{flatten}"`

這僅執行 `aiken/option` 模組中名稱包含單詞 `flatten` 的測試。

---

`aiken check -e -m "aiken/option.{flatten_1}"`

您可以使用 `-e` 強制精確匹配。

---

`aiken check -e -m map_1`

這僅執行整個專案中與名稱 `map_1` 精確匹配的測試。
