# EUTxO 速成課程

如果您完全不知道在 Cardano 上開發是什麼樣子，別擔心。您剛剛找到了最適合入門的內容。Aiken 是一個讓鏈上編寫程式變得簡單的語言平台。但首先，什麼是「鏈上編寫程式」？雖然這篇簡短的文件並非旨在成為完整的區塊鏈課程，但它應該能給您足夠的見解，以建立對基礎知識的基本瞭解。

  本課程將引用密碼學概念，例如 **哈希摘要 (hash digests)** 或 **數位簽章 (digital signatures)**。因此，我們希望讀者能熟悉這些概念（至少了解一點點），或者自行查閱相關資料。網路上有大量關於密碼學的資源，而本速成課程 **並非** 其中之一。

  作為建議，如果您想瞭解更多基礎知識，我們由衷推薦 [Cardano 區塊鏈認證助理 (CBCA) 的第一個模組](https://cardanofoundation.org/en/academy/course/cbca)。
</Callout>

## 區塊與交易

區塊鏈由區塊組成。而區塊由交易組成。不深入細節的話，您可以將區塊視為分為兩個部分的物件：區塊頭 (header) 和區塊體 (body)。區塊頭包含有關區塊的資訊，例如誰生產了區塊以及何時生產。區塊體則純粹是一個有序的交易序列。

請注意，區塊鏈中的「鏈」來自於區塊如何相互引用。事實上，每個區塊頭至少包含兩樣東西：

- 區塊體的哈希摘要
- 前一個區塊頭的哈希摘要

```console filename="Block"
┏━ Header ━━━━━━━━━━━━━━┳━ Body ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃                       ┃                                           ┃
┃  Body hash            ┃ ┌────────────────┬────────────────┬─────┐ ┃
┃  Previous header hash ┃ │ Transaction #1 │ Transaction #2 │ ... │ ┃
┃  Timestamp            ┃ └────────────────┴────────────────┴─────┘ ┃
┃                       ┃                                           ┃
┗━━━━━━━━━━━━━━━━━━━━━━━┻━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
```

哈希摘要是一種防篡改機制，它將輸入值映射為固定大小的輸出。可以將其視為為一段內容分配標識符的方法，標識符取決於內容本身：內容改變，標識符也會隨之改變。

區塊鏈透過在每個區塊頭中包含以下內容來形成鏈條：

- 區塊體的哈希；以及
- 前一個區塊的區塊頭哈希。

更改區塊中的任何交易都會更改區塊體哈希，進而更改該區塊的區塊頭哈希、下一個區塊的區塊頭哈希，依此類推，導致後續整條鏈失效。

```
        ____          ____          ____          ____          ____
       /    /\       /    /\       /    /\       /    /\       /    /\
o ❮❮  /____/  \ ❮❮  /____/  \ ❮❮  /____/  \ ❮❮  /____/  \ ❮❮  /____/  \ ...
      \    \  /     \    \  /     \    \  /     \    \  /     \    \  /
╿      \____\/       \____\/       \____\/       \____\/       \____\/
│
│                                     ╿
│                                     │
└ Genesis configuration               │
                                      └ Block
```

因此，交易是區塊鏈上最基本的原語。它是使用者（也就是您）可以採取行動來更改區塊鏈狀態的機制。一條鏈從初始狀態開始，通常稱為 *創世配置 (genesis configuration)*。從那裡開始，交易將前一個狀態映射為新狀態。最後，區塊僅僅是為了將交易批次處理在一起。

## 未花費交易輸出 (UTxO)

在傳統資料庫世界中，「交易」是將一系列原子操作捆綁在一起的一種手段，因此要麼全部成功，要麼都不會發生。在金融世界中，它是將資產從一個位置轉移到另一個位置的一種方法。

在區塊鏈世界中，它兩者兼具。

交易首先是一個物件，它具有資產來源的「輸入」和資產去向的「輸出」。通常，正如 Cardano 的情況，交易具有多個輸入和多個輸出。此外，除了輸入和輸出，區塊鏈協議通常還包括修改區塊鏈狀態不同部分的其他元素（例如委託憑證、治理投票、使用者定義的資產定義...）。

更重要的是，就像資料庫世界一樣，交易是一系列全有或全無的原子命令。它要麼有效且套用其所有更改，要麼無效且不套用任何更改。

我們稍後會討論其他功能。現在，讓我們專注於輸入和輸出，首先從輸出開始。

### 輸出

在 Cardano 中，一個輸出是一個描述至少兩件事的物件：

- 資產數量 —— 也稱為「價值」(Value)；
- 花費（和/或委託）這些資產的條件 —— 也稱為「地址」(Address)。

此外，資料載荷 (datum) 也可以新增到輸出中，但現在先別管它。價值的角色非常透明，它指示輸出持有多少資產。

順帶一提，Cardano 支援兩種資產：主要協議貨幣（即 Ada）和使用者定義貨幣。兩者都共存於價值中，儘管套用於各自的規則略有不同。

地址擷取了告知協議在什麼條件下可以使用特定輸出的資產的邏輯。它是定義資產所有權的東西。我們很快就會探究這一點。請再耐心等一下。

```console filename="Output"
┃         ┃           ┃                ┃
┃  Value  ┃  Address  ┃  Data payload  ┃
┃         ┃           ┃                ┃

```

### 輸入

輸入是對前一個輸出的引用。將輸出想像成帶有唯一序列號的便利貼，而輸入就是這個序列號。

![](/post-it-notes.jpg)

交易是一份文件，指示哪些便利貼應該被銷毀，以及哪些新的便利貼應該被貼在牆上。請注意，關於交易的建構是有規則的。例如，輸入的總價值必須等於輸出的總價值。換句話說，總價值應該平衡，但可能會重新分配。

尚未花費的輸出（即仍在牆上的內容）被稱為 —— 您猜對了 —— **未花費交易輸出 (unspent transaction output)**，簡稱 UTxO。區塊鏈狀態是透過查看剩餘便利貼的整面牆而產生的。除了所有可用的 UTxO，狀態還包括協議定義的任何額外數據。

好，回到輸入。

技術上來說，一個輸入的「序列號」是發出它所引用的輸出的交易哈希摘要，以及該輸出在該交易中的位置。這兩個元素使得每個輸入都是唯一的。而且因為輸出在花費時會從可用集合中移除（便利貼被銷毀），所以它們只能被花費一次。這由區塊鏈協議確保。

```console filename="Input"
┏━━━━━━━━━━━━━━━━━━━━┳━━━━━━━━━━━━━━━━┓
┃                    ┃                ┃
┃  Transaction hash  ┃  Output index  ┃
┃                    ┃                ┃
┗━━━━━━━━━━━━━━━━━━━━┻━━━━━━━━━━━━━━━━┛
```

<Callout type="info">
第一個輸出是從哪裡來的？

如果您仔細跟隨了我們剛才的敘述，您可能已經發現我們遇到了雞生蛋、蛋生雞的問題。輸入是對輸出的引用，而輸出是透過花費輸入建立的。

這就是創世配置的用途。它以一致同意的初始輸出列表的形式定義了區塊鏈的起點。那些輸出可以使用某些特殊標識符來引用。例如，創世配置哈希摘要和輸出在配置中的位置。

創世配置是如何產生的也是一個有趣的問題，但超出了目前課程的範圍。以 Cardano 為例，初始分配源於初始代幣憑證銷售，在網路啟動前，一部分 Ada 總供應量以憑證的形式出售給了利害關係人。
</Callout>

### TL;DR

讓我們快速回顧一下到目前為止我們所看到的：

- 區塊鏈有一個初始狀態，稱為 *創世配置*；
- *交易* 擷取了修改該狀態的指令（例如資產轉移）；
- *區塊* 將交易批次處理在一起，並具有對父區塊的引用；
- 資產移動在交易中使用 *輸入* 和 *輸出* 來表達；
- *輸出* 是一個至少具有 *地址* 和 *價值* 的物件；
- *地址* 描述了使用與其相關聯的 *價值* 所需的條件；
- *輸入* 是對前一個 *輸出* 的引用。

## 地址

### 概覽

現在是時候深入探究 Cardano 地址了。一個典型的地址由 2 或 3 個部分組成：

```console filename="Address"
┏━━━━━━━━━━┳━━━━━━━━━━━━━━━━━━━━━━━┳━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃          ┃                       ┃                          ┃
┃  Header  ┃  Payment credentials  ┃  Delegation credentials  ┃
┃          ┃                       ┃                          ┃
┗━━━━━━━━━━┻━━━━━━━━━━━━━━━━━━━━━━━┻━━━━━━━━━━━━━━━━━━━━━━━━━━┛
```

我們說 2 或 3 個部分是因為最後一部分 —— 委託憑證 —— 是選填的。第一部分稱為 `Header`，它描述了地址類型（即接下來是什麼內容）以及該地址可以在哪個網路中使用。我們將最後一小部分稱為 *網路辨別碼 (network discriminant)*，它可以防止諸如將真正的 Mainnet 資金發送到測試地址之類的愚蠢錯誤。地址表示為位元組序列，通常使用 [bech32](https://github.com/bitcoin/bips/blob/master/bip-0173.mediawiki#user-content-Introduction) 或簡單的 [base16](https://en.wikipedia.org/wiki/Hexadecimal) 文本字串編碼。

例如：

```console filename="An address (bech32)"
addr1x8phkx6acpnf78fuvxn0mkew3l0fd058hzquvz7w36x4gt7r0vd4msrxnuwnccdxlhdjar77j6lg0wypcc9uar5d2shskhj42g
```

或者：

```console filename="The same address (base16)"
31c37b1b5dc0669f1d3c61a6fddb2e8fde96be87b881c60bce8e8d542fc37b1b5dc0669f1d3c61a6fddb2e8fde96be87b881c60bce8e8d542f
```

我們可以分解後者的文本字串，使上述三個部分更加明顯：

```
               Type = 3 ┐┌ Network = 1 (1 = mainnet, 0 = testnet)
                        ││
                        ╽╽
Header:                 31
Payment credentials:    c37b1b5dc0669f1d3c61a6fddb2e8fde96be87b881c60bce8e8d542f
Delegation credentials: c37b1b5dc0669f1d3c61a6fddb2e8fde96be87b881c60bce8e8d542f
```

如我們所見，該地址是類型 3，用於主網，並且支付和委託部分使用相同的憑證。

<Callout type="info">
  您經常需要在 bech32 編碼字串和 hex 編碼字串之間來回轉換。工作中處理 bech32 的一個很棒的命令行工具可以在 [input-output-hk/bech32](https://github.com/input-output-hk/bech32#bech32-command-line) 找到。儘管使用它吧！
</Callout>

### 支付憑證

下一部分是 *支付憑證 (payment credentials)*，也稱為 *支付部分*。這是描述地址花費條件的部分。還記得 UTxO 就像牆上的便利貼嗎？嗯，您不能直接自己貼上去或撕下來。您必須向網路驗證者提交一份 *交易*。想像一個看管便利貼牆的員工，您必須給他一張描述您想做什麼的表格。每張便利貼上都寫著在被撕下並銷毀之前必須滿足的條件。這就是地址中支付憑證的用途。它們以兩種形式之一出現：

- 驗證金鑰哈希摘要；或
- 腳本哈希摘要。

在第一種形式中，驗證者節點 —— 或員工 —— 會要求您提供與驗證金鑰相對應的簽署金鑰的數位簽章。這種方法依賴於非對稱加密，其中會生成一對公鑰（驗證）和私鑰（簽署）。在地址中，我們僅儲存驗證金鑰的哈希摘要，以保證簡潔並避免過早揭露它（儘管它是公開材料）。當從此類地址花費時，必須揭露公鑰並出示整個交易的簽章作為 *見證人 (witnesses)*（亦稱證明）。這種指定花費條件的方式相對簡單，但也受到限制，因為它不允許表達任何複雜的邏輯。

這就是第二種形式變得更有趣的地方。Cardano 允許使用代表驗證邏輯的腳本來鎖定資金，在花費該地址守衛的資金時必須滿足該邏輯。我們通常稱此類地址為「腳本地址」。與第一種形式類似，任何從腳本地址花費的交易都必須提供整個腳本作為見證，以及腳本所需的任何其他元素。腳本就像謂詞 (predicates)。換句話說，它們是返回布林值的函數：`True` 或 `False`。要被視為有效，交易中的所有腳本都必須返回 `True`。我們稍後將探究此機制是如何運作的。

### 委託憑證

地址也可以包含 *委託憑證 (delegation credentials)*，也稱為 *委託部分*。我們這裡只做簡要介紹，您可以將委託憑證視為一種控制如何處理與地址相關聯的「權益」(stake) 的方法。「權益」對應於輸出價值中 Ada 的數量，共識協議會計算這些數量來選舉區塊生產者。在 Cardano 中，權益可以委託給稱為權益池 (stake pools) 的註冊實體。透過委託，表示與輸出相關聯的權益應視為屬於被委託者，從而增加他們生產區塊的機會。作為回報，被委託者同意與委託者分享其生產區塊的部分獎勵。

支付憑證控制如何 **花費** 輸出，而委託憑證則控制兩個獨立的操作：

- 如何 **發布** 委託憑證（例如，將權益委託給權益池）；
- 如何 **提領** 與權益憑證相關聯的獎勵。

與支付憑證類似，委託憑證也有兩種形式：驗證金鑰哈希摘要或腳本哈希摘要。

</Callout>

在繼續之前，讓我們再次回顧一下：

- 一個 *地址* 由 2 或 3 個部分組成：它總是包含 *標頭* 和 *支付憑證*，並且可以選填地包含 *委託憑證*；
- *標頭* 描述地址 *類型* 及其所屬的 *網路*；
- *委託憑證* 部分雖然是選填的，但仍然 <u>極力推薦</u>；
- 憑證（支付或委託）採用以下兩種形式之一：
  - 驗證金鑰哈希；
  - 腳本哈希；
- 支付憑證控制如何從 *地址* **花費**；
- 委託憑證控制如何 **發布** 憑證以及如何 **提領** 獎勵；
- 腳本允許定義任意的驗證邏輯。

## 腳本、Datum 與 Redeemer

### 概覽

堅持住！我們快要完成這個速成課程了。我們已經瞭解了 UTxO 是什麼以及地址是如何構成的，並且我們討論了一點腳本。特別是，我們說過腳本就像謂詞：也就是說，是以交易為參數並返回 `True` 或 `False` 的純函數（在數學意義上）。

嗯，*不完全是*。我們對您撒了個小慌（但只有一點點）。

如果我們只有這些，就很難表達更複雜的邏輯。特別是，程式通常需要的狀態捕捉將是不可行的 —— 一種狀態以及從該狀態過渡的過程。這就是 *擴充 UTxO (extended UTxO)* 模型的用武之地。它在我們已經看到的內容基礎上增加了兩個新組件：Datum 和 Redeemer。

我們之前提到過 Datum（當時沒叫它 Datum），當時我們說輸出包含價值、地址和資料載荷。這就是 Datum 的本質，它是開發者可用於在腳本執行時附加數據的自由載荷。當腳本在花費場景中執行時，它不僅接收交易作為上下文，還接收與被花費輸出相關聯的 Datum。

另一方面，Redeemer 是另一段數據，同樣隨交易一起提供給任何腳本執行。請注意，Datum 和 Redeemer 在兩個不同的時刻介入。Datum 是在建立輸出時設置的（即當便利貼被貼在牆上時，它是貼紙的一部分）。相比之下，Redeemer 僅在花費輸出時提供（即在將表單交給員工時隨表單一起提供）。

### 比喻

另一種看待腳本、Datum 和 Redeemer 的方式是將它們視為共同構成了帶參數的數學函數。

```console filename="Script, datum and redeemer"
             Script
          ╭─────────╮
    f(x) = x * a + b  = true | false
           ╿   ╿   ╿
  Redeemer ┘   │   │
               └─┬─┘
               Datum
```

腳本定義了整個函數。它指示參數和引數如何組合以產生布林結果。Datum 對應於函數的參數。它允許 *配置函數* 並使用不同的設置重複使用類似的驗證邏輯。函數和參數都是在資產被鎖定在輸出中時定義的。這僅留下稍後提供的函數引數。該引數就是 Redeemer（以及交易的其餘部分）。

這就是為什麼腳本通常被稱為 *驗證器 (validators)*。因此，與某些其他區塊鏈系統不同，它們是完全確定性的。它們的執行僅取決於它們參與的交易，並且在將交易發送到網路之前，評估交易的結果是可能的。Datum 充當局部狀態，而 Redeemer 是在交易本身中提供的使用者輸入。

<Callout type="info">
  如果我們退一步看看花費資金的典型公鑰/私鑰過程，我們可以看到 eUTxO 僅僅是其一般化。事實上，公鑰（哈希）可以被視為 *Datum*，而簽名則是 *Redeemer*。腳本則是數位簽章驗證演算法，它控制簽名相對於所提供金鑰是否有效。
</Callout>

### 腳本用途 (Purposes)

到目前為止，我們主要是在驗證輸出是否可以被花費的上下文中討論腳本。我們之前也簡要提到過如何使用腳本來控制委託憑證的發布或如何提領共識獎勵。

這些不同的使用案例通常被稱為「腳本用途」(script purposes)。到目前為止，我們已經看到了三種用途：`spend` (花費)、`publish` (發布) 和 `withdraw` (提領)。還有另外三種：`mint` (鑄造)、`vote` (投票) 和 `propose` (提案)。

每種用途都指示了腳本執行是 *為了什麼目的*。在驗證期間，該資訊隨交易和 Redeemer 一起傳遞給腳本。請注意，只有以 `spend` 用途執行的腳本才會獲得 Datum。這是因為它們可以利用輸出中存在的資料載荷，而其他用途則沒有這個機會。

#### 鑄造 (Mint)

鑄造用途是指為了驗證是否可以鑄造（即建立）或銷毀（即燒毀）使用者定義的資產而執行的腳本。Cardano 確實支援使用者定義的資產，這些資產用於表示同質化數量（如協議貨幣）或非同質化數量（即 NFT）。

管理資產建立或銷毀的規則被定義為腳本。我們通常將此類腳本稱為 *鑄造策略 (minting policies)*，對應於上述的 `mint` 用途。

#### 投票 (Vote)

投票用途驗證來自腳本委託代表的治理投票。每個交易中每個委託憑證執行一次，即使該交易實際上包含多次投票。這便於構建比純公鑰更複雜的委託代表；例如，模擬冷/熱鑰設置。

#### 提案 (Propose)

最後，提案用途是非常獨特的一種，因為整個帳本中只能有一個腳本可以執行該用途：憲法護欄腳本 (constitution guardrails script)。該腳本是隨鏈上憲法（選填地）定義的，由帳本為每個提議治理行動的交易執行。其目標是確保某些提案從一開始就被拒絕，以防止網路中出現考慮周詳的提案產生的垃圾訊息 —— 特別是在協議參數方面。

例如，一項將區塊大小從當前值降低到零的提案將完全使鏈條癱瘓，而且無論如何都不太可能被接受。護欄腳本可以防止這類提案被提交。原則上，它是憲法的程式化執行。

### TL;DR

我們已經來到了速成課程的終點。讓我們最後對腳本、Datum 和 Redeemer 進行總結。

- *腳本 (Scripts)* 類似於帶參數的謂詞函數，返回 true 或 false。
- *Datum* 充當函數參數的角色，而 *Redeemer* 則充當引數的角色。
- *腳本* 也被稱為 *驗證器 (validators)* 並且是完全確定性的。
- *腳本* 用於驗證交易中的特定操作。
- *腳本* 的用途被稱為其 *Purpose*。共有 6 種用途：
  - `mint` —— 控制如何鑄造或銷毀資產；
  - `spend` —— 控制如何花費輸出；
  - `withdraw` —— 控制如何提領共識獎勵；
  - `publish` —— 控制如何發布委託憑證；
  - `vote` —— 控制如何對提案程序進行投票；
  - `propose` —— 控制提案程序的憲法性。
- 只有 *花費腳本*（即 purpose=spend）可以訪問 Datum。
