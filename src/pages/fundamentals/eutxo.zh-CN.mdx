# EUTxO 速成课程

如果您完全不知道在 Cardano 上开发是什么样子，别担心。您刚刚找到了最适合入门的内容。Aiken 是一个让链上编写程序变得简单的语言平台。但首先，什么是「链上编写程序」？虽然这篇简短的文档并非旨在成为完整的区块链课程，但它应该能给您足够的见解，以建立对基础知识的基本了解。

  本课程将引用密码学概念，例如 **哈希摘要 (hash digests)** 或 **数字签名 (digital signatures)**。因此，我们希望读者能熟悉这些概念（至少了解一点点），或者自行查阅相关资料。网络上有大量关于密码学的资源，而本速成课程 **并非** 其中之一。

  作为建议，如果您想了解更多基础知识，我们由衷推荐 [Cardano 区块链认证助理 (CBCA) 的第一个模块](https://cardanofoundation.org/en/academy/course/cbca)。
</Callout>

## 区块与交易

区块链由区块组成。而区块由交易组成。不深入细节的话，您可以将区块视为分为两个部分的物件：区块头 (header) 和区块体 (body)。区块头包含有关区块的信息，例如谁生产了区块以及何时生产。区块体则纯粹是一个有序的交易序列。

请注意，区块链中的「链」来自区块如何相互引用。事实上，每个区块头至少包含两样东西：

- 区块体的哈希摘要
- 前一个区块头的哈希摘要

```console filename="Block"
┏━ Header ━━━━━━━━━━━━━━┳━ Body ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃                       ┃                                           ┃
┃  Body hash            ┃ ┌────────────────┬────────────────┬─────┐ ┃
┃  Previous header hash ┃ │ Transaction #1 │ Transaction #2 │ ... │ ┃
┃  Timestamp            ┃ └────────────────┴────────────────┴─────┘ ┃
┃                       ┃                                           ┃
┗━━━━━━━━━━━━━━━━━━━━━━━┻━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
```

哈希摘要是一种防篡改机制，它将输入值映射为固定大小的输出。可以将其视为为一段内容分配标识符的方法，标识符取决于内容本身：内容改变，标识符也会随之改变。

区块链通过在每个区块头中包含以下内容来形成链条：

- 区块体的哈希；以及
- 前一个区块的区块头哈希。

更改区块中的任何交易都会更改区块体哈希，从而更改该区块的区块头哈希、下一个区块的区块头哈希，依此类推，导致后续整条链失效。

```
        ____          ____          ____          ____          ____
       /    /\       /    /\       /    /\       /    /\       /    /\
o ❮❮  /____/  \ ❮❮  /____/  \ ❮❮  /____/  \ ❮❮  /____/  \ ❮❮  /____/  \ ...
      \    \  /     \    \  /     \    \  /     \    \  /     \    \  /
╿      \____\/       \____\/       \____\/       \____\/       \____\/
│
│                                     ╿
│                                     │
└ Genesis configuration               │
                                      └ Block
```

因此，交易是区块链上最基本的原语。它是用户（也就是您）可以采取行动来更改区块链状态的机制。一条链从初始状态开始，通常称为 *创世配置 (genesis configuration)*。从那里开始，交易将前一个状态映射为新状态。最后，区块仅仅是为了将交易批次处理在一起。

## 未花费交易输出 (UTxO)

在传统数据库世界中，「交易」是将一系列原子操作捆绑在一起的一种手段，因此要么全部成功，要么都不会发生。在金融世界中，它是将资产从一个位置转移到另一个位置的一种方法。

在区块链世界中，它两者兼具。

交易首先是一个物件，它具有资产来源的「输入」和资产去向的「输出」。通常，正如 Cardano 的情况，交易具有多个输入和多个输出。此外，除了输入和输出，区块链协议通常还包括修改区块链状态不同部分的其他元素（例如委托凭证、治理投票、用户定义的资产定义...）。

更重要的是，就像数据库世界一样，交易是一系列全有或全无的原子命令。它要么有效且应用其所有更改，要么无效且不应用任何更改。

我们稍后会讨论其他功能。现在，让我们专注于输入和输出，首先从输出开始。

### 输出

在 Cardano 中，一个输出是一个描述至少两件事的物件：

- 资产数量 —— 也称为「价值」(Value)；
- 花费（和/或委托）这些资产的条件 —— 也称为「地址」(Address)。

此外，数据载荷 (datum) 也可以添加到输出中，但现在先别管它。价值的角色非常透明，它指示输出持有多少资产。

顺带一提，Cardano 支持两种资产：主要协议货币（即 Ada）和用户定义货币。两者都共存于价值中，尽管应用于各自的规则略有不同。

地址捕获了告知协议在什么条件下可以使用特定输出的资产的逻辑。它是定义资产所有权的东西。我们很快就会探究这一点。请再耐心等一下。

```console filename="Output"
┏━━━━━━━━━┳━━━━━━━━━━━┳━━━━━━━━━━━━━━━━┓
┃         ┃           ┃                ┃
┃  Value  ┃  Address  ┃  Data payload  ┃
┃         ┃           ┃                ┃
┗━━━━━━━━━┻━━━━━━━━━━━┻━━━━━━━━━━━━━━━━┛
```

### 输入

输入是对前一个输出的引用。将输出想象成带有唯一序列号的便利贴，而输入就是这个序列号。

![](/post-it-notes.jpg)

交易是一份文档，指示哪些便利贴应该被销毁，以及哪些新的便利贴应该被贴在墙上。请注意，关于交易的构建是有规则的。例如，输入的总价值必须等于输出的总价值。换句话说，总价值应该平衡，但可能会重新分配。

尚未花费的输出（即仍在墙上的内容）被称为 —— 您猜对了 —— **未花费交易输出 (unspent transaction output)**，简称 UTxO。区块链状态是通过查看剩余便利贴的整面墙而产生的。除了所有可用的 UTxO，状态还包括协议定义的任何额外数据。

好，回到输入。

技术上来说，一个输入的「序列号」是发出它所引用的输出的交易哈希摘要，以及该输出在该交易中的位置。这两个元素使得每个输入都是唯一的。而且因为输出在花费时会从可用集合中移除（便利贴被销毁），所以它们只能被花费一次。这由区块链协议确保。

```console filename="Input"
┏━━━━━━━━━━━━━━━━━━━━┳━━━━━━━━━━━━━━━━┓
┃                    ┃                ┃
┃  Transaction hash  ┃  Output index  ┃
┃                    ┃                ┃
┗━━━━━━━━━━━━━━━━━━━━┻━━━━━━━━━━━━━━━━┛
```

<Callout type="info">
第一个输出是从哪里来的？

如果您仔细跟随了我们刚才的叙述，您可能已经发现我们遇到了鸡生蛋、蛋生鸡的问题。输入是对输出的引用，而输出是通过花费输入创建的。

这就是创世配置的用途。它以一致同意的初始输出列表的形式定义了区块链的起点。那些输出可以使用某些特殊标识符来引用。例如，创世配置哈希摘要和输出在配置中的位置。

创世配置是如何产生的也是一个有趣的问题，但超出了目前课程的范围。以 Cardano 为例，初始分配源于初始代币凭证销售，在网络启动前，一部分 Ada 总供应量以凭证的形式出售给了利益相关人。
</Callout>

### TL;DR

让我们快速回顾一下到目前为止我们所看到的：

- 区块链有一个初始状态，称为 *创世配置*；
- *交易* 捕获了修改该状态的指令（例如资产转移）；
- *区块* 将交易批次处理在一起，并具有对父区块的引用；
- 资产移动在交易中使用 *输入* 和 *输出* 来表达；
- *输出* 是一个至少具有 *地址* 和 *价值* 的物件；
- *地址* 描述了使用与其相关联的 *价值* 所需的条件；
- *输入* 是对前一个 *输出* 的引用。

## 地址

### 概览

现在是时候深入探究 Cardano 地址了。一个典型的地址由 2 或 3 个部分组成：

```console filename="Address"
┏━━━━━━━━━━┳━━━━━━━━━━━━━━━━━━━━━━━┳━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃          ┃                       ┃                          ┃
┃  Header  ┃  Payment credentials  ┃  Delegation credentials  ┃
┃          ┃                       ┃                          ┃
┗━━━━━━━━━━┻━━━━━━━━━━━━━━━━━━━━━━━┻━━━━━━━━━━━━━━━━━━━━━━━━━━┛
```

我们说 2 或 3 个部分是因为最后一部分 —— 委托凭证 —— 是选填的。第一部分称为 `Header`，它描述了地址类型（即接下来是什么内容）以及该地址可以在哪个网络中使用。我们将最后一小部分称为 *网络辨别码 (network discriminant)*，它可以防止诸如将真正的 Mainnet 资金发送到测试地址之类的愚蠢错误。地址表示为字节序列，通常使用 [bech32](https://github.com/bitcoin/bips/blob/master/bip-0173.mediawiki#user-content-Introduction) 或简单的 [base16](https://en.wikipedia.org/wiki/Hexadecimal) 文本字符串编码。

例如：

```console filename="An address (bech32)"
addr1x8phkx6acpnf78fuvxn0mkew3l0fd058hzquvz7w36x4gt7r0vd4msrxnuwnccdxlhdjar77j6lg0wypcc9uar5d2shskhj42g
```

或者：

```console filename="The same address (base16)"
31c37b1b5dc0669f1d3c61a6fddb2e8fde96be87b881c60bce8e8d542fc37b1b5dc0669f1d3c61a6fddb2e8fde96be87b881c60bce8e8d542f
```

我们可以分解后者的文本字符串，使上述三个部分更加明显：

```
               Type = 3 ┐┌ Network = 1 (1 = mainnet, 0 = testnet)
                        ││
                        ╽╽
Header:                 31
Payment credentials:    c37b1b5dc0669f1d3c61a6fddb2e8fde96be87b881c60bce8e8d542f
Delegation credentials: c37b1b5dc0669f1d3c61a6fddb2e8fde96be87b881c60bce8e8d542f
```

如我们所见，该地址是类型 3，用于主网，并且支付和委托部分使用相同的凭证。

<Callout type="info">
  您经常需要在 bech32 编码字符串和 hex 编码字符串之间来回转换。工作中处理 bech32 的一个很棒的命令行工具可以在 [input-output-hk/bech32](https://github.com/input-output-hk/bech32#bech32-command-line) 找到。尽管使用它吧！
</Callout>

### 支付凭证

下一部分是 *支付凭证 (payment credentials)*，也称为 *支付部分*。这是描述地址花费条件的部分。还记得 UTxO 就像墙上的便利贴吗？嗯，您不能直接自己贴上去或撕下来。您必须向网络验证者提交一份 *交易*。想象一个看管便利贴墙的员工，您必须给他一张描述您想做的表格。每张便利贴上都写着在被撕下并销毁之前必须满足的条件。这就是地址中支付凭证的用途。它们以两种形式之一出现：

- 验证密钥哈希摘要；或
- 脚本哈希摘要。

在第一种形式中，验证者节点 —— 或员工 —— 会要求您提供与验证密钥相对应的签名密钥的数字签名。这种方法依赖于非对称加密，其中会生成一对公钥（验证）和私钥（签名）。在地址中，我们仅存储验证密钥的哈希摘要，以保证简洁并避免过早揭露它（尽管它是公开材料）。当从此类地址花费时，必须揭露公钥并出示整个交易的签名作为 *见证人 (witnesses)*（亦称证明）。这种指定花费条件的方式相对简单，但也受到限制，因为它不允许表达任何复杂的逻辑。

这就是第二种形式变得更有趣的地方。Cardano 允许使用代表验证逻辑的脚本来锁定资金，在花费该地址守卫的资金时必须满足该逻辑。我们通常称此类地址为「脚本地址」。与第一种形式类似，任何从脚本地址花费的交易都必须提供整个脚本作为见证，以及脚本所需的任何其他元素。脚本就像谓词 (predicates)。换句话说，它们是返回布尔值的函数：`True` 或 `False`。要被视为有效，交易中的所有脚本都必须返回 `True`。我们稍后将探究此机制是如何运作的。

### 委托凭证

地址也可以包含 *委托凭证 (delegation credentials)*，也称为 *委托部分*。我们这里只做简要介绍，您可以将委托凭证视为一种控制如何处理与地址相关联的「权益」(stake) 的方法。「权益」对应于输出价值中 Ada 的数量，共识协议会计算这些数量来选举区块生产者。在 Cardano 中，权益可以委托给称为权益池 (stake pools) 的注册实体。通过委托，表示与输出相关联的权益应视为属于被委托者，从而增加他们生产区块的机会。作为回报，被委托者同意与委托者分享其生产区块的部分奖励。

支付凭证控制如何 **花费** 输出，而委托凭证则控制两个独立的操作：

- 如何 **发布** 委托凭证（例如，将权益委托给权益池）；
- 如何 **提领** 与权益凭证相关联的奖励。

与支付凭证类似，委托凭证也有两种形式：验证密钥哈希摘要或脚本哈希摘要。

<Callout type="info">
  有关地址及其工作原理的更多信息，请参见 [CIP-0019](https://github.com/cardano-foundation/CIPs/tree/master/CIP-0019/#readme)
</Callout>

在继续之前，让我们再次回顾一下：

- 一个 *地址* 由 2 或 3 个部分组成：它总是包含 *标头* 和 *支付凭證*，并且可以选填地包含 *委托凭证*；
- *标头* 描述地址 *类型* 及其所属的 *网络*；
- *委托凭证* 部分虽然是选填的，但仍然 <u>极力推荐</u>；
- 凭证（支付或委托）采用以下两种形式之一：
  - 验证密钥哈希；
  - 脚本哈希；
- 支付凭证控制如何从 *地址* **花费**；
- 委托凭证控制如何 **发布** 凭证以及如何 **提领** 奖励；
- 脚本允许定义任意的验证逻辑。

## 脚本、Datum 与 Redeemer

### 概览

坚持住！我们快要完成这个速成课程了。我们已经了解了 UTxO 是什么以及地址是如何构成的，并且我们讨论了一点脚本。特别是，我们说过脚本就像谓词：也就是说，是以交易为参数并返回 `True` 或 `False` 的纯函数（在数学意义上）。

嗯，*不完全是*。我们对您撒了个小谎（但只有一点点）。

如果我们只有这些，就很难表达更复杂的逻辑。特别是，程序通常需要的状态捕获将是不可行的 —— 一种状态以及从该状态过渡的过程。这就是 *扩展 UTxO (extended UTxO)* 模型的用武之地。它在我们已经看到的内容基础上增加了两个新组件：Datum 和 Redeemer。

我们之前提到过 Datum（当时没叫它 Datum），当时我们说输出包含价值、地址和数据载荷。这就是 Datum 的本质，它是开发者可用于在脚本执行时附加数据的自由载荷。当脚本在花费场景中执行时，它不仅接收交易作为上下文，还接收与被花费输出相关联的 Datum。

另一方面，Redeemer 是另一段数据，同样随交易一起提供给任何脚本执行。请注意，Datum 和 Redeemer 在两个不同的时刻介入。Datum 是在创建输出时设置的（即当便利贴被贴在墙上时，它是贴纸的一部分）。相比之下，Redeemer 仅在花费输出时提供（即在将表单交给员工时随表单一起提供）。

### 比喻

另一种看待脚本、Datum 和 Redeemer 的方式是将它们视为共同构成了带参数的数学函数。

```console filename="Script, datum and redeemer"
             Script
          ╭─────────╮
    f(x) = x * a + b  = true | false
           ╿   ╿   ╿
  Redeemer ┘   │   │
               └─┬─┘
               Datum
```

脚本定义了整个函数。它指示参数和实参如何组合以产生布尔结果。Datum 对应于函数的参数。它允许 *配置函数* 并使用不同的设置重复使用类似的验证逻辑。函数和参数都是在资产被锁定在输出中时定义的。这仅留下稍后提供的函数实参。该实参就是 Redeemer（以及交易的其余部分）。

这就是为什么脚本通常被称为 *验证器 (validators)*。因此，与某些其他区块链系统不同，它们是完全确定性的。它们的执行仅取决于它们参与的交易，并且在将交易发送到网络之前，评估交易的结果是可能的。Datum 充当局部状态，而 Redeemer 是在交易本身中提供的用户输入。

<Callout type="info">
  如果我们退一步看看花费资金的典型公钥/私钥过程，我们可以看到 eUTxO 仅仅是其一般化。事实上，公钥（哈希）可以被视为 *Datum*，而签名则是 *Redeemer*。脚本则是数字签名验证算法，它控制签名相对于所提供密钥是否有效。
</Callout>

### 脚本用途 (Purposes)

到目前为止，我们主要是在验证输出是否可以被花费的上下文中讨论脚本。我们之前也简要提到过如何使用脚本来控制委托凭证的发布或如何提领共识奖励。

这些不同的使用案例通常被称为「脚本用途」(script purposes)。到目前为止，我们已经看到了三种用途：`spend` (花费)、`publish` (发布) 和 `withdraw` (提领)。还有另外三种：`mint` (铸造)、`vote` (投票) 和 `propose` (提案)。

每种用途都指示了脚本执行是 *为了什么目的*。在验证期间，该信息随交易和 Redeemer 一起传递给脚本。请注意，只有以 `spend` 用途执行的脚本才会获得 Datum。这是因为它们可以利用输出中存在的数据载荷，而其他用途则没有这个机会。

#### 铸造 (Mint)

铸造用途是指为了验证是否可以铸造（即创建）或销毁（即烧毁）用户定义的资产而执行的脚本。Cardano 确实支持用户定义的资产，这些资产用于表示同质化数量（如协议货币）或非同质化数量（即 NFT）。

管理资产创建或销毁的规则被定义为脚本。我们通常将此类脚本称为 *铸造策略 (minting policies)*，对应于上述的 `mint` 用途。

#### 投票 (Vote)

投票用途验证来自脚本委托代表的治理投票。每个交易中每个委托凭证执行一次，即使该交易实际上包含多次投票。这便于构建比纯公钥更复杂的委托代表；例如，模拟冷/热密钥设置。

#### 提案 (Propose)

最后，提案用途是非常独特的一种，因为整个账本中只能有一个脚本可以执行该用途：宪法护栏脚本 (constitution guardrails script)。该脚本是随链上宪法（选填地）定义的，由账本为每个提议治理行动的交易执行。其目标是确保某些提案从一开始就被拒绝，以防止网络中出现考虑周详的提案产生的垃圾信息 —— 特别是在协议参数方面。

例如，一项将区块大小从当前值降低到零的提案将完全使链条瘫痪，而且无论如何都不太可能被接受。护栏脚本可以防止这类提案被提交。原则上，它是宪法的程式化执行。

### TL;DR

我们已经来到了速成课程的终点。让我们最后对脚本、Datum 和 Redeemer 进行总结。

- *脚本 (Scripts)* 类似于带参数的谓词函数，返回 true 或 false。
- *Datum* 充当函数参数的角色，而 *Redeemer* 则充当实参的角色。
- *脚本* 也被称为 *验证器 (validators)* 并且是完全确定性的。
- *脚本* 用于验证交易中的特定操作。
- *脚本* 的用途被称为其 *Purpose*。共有 6 种用途：
  - `mint` —— 控制如何铸造或销毁资产；
  - `spend` —— 控制如何花费输出；
  - `withdraw` —— 控制如何提领共识奖励；
  - `publish` —— 控制如何发布委托凭证；
  - `vote` —— 控制如何对提案程序进行投票；
  - `propose` —— 控制提案程序的宪法性。
- 只有 *花费脚本*（即 purpose=spend）可以访问 Datum。
