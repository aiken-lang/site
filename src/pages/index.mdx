<details open>
  <summary>
    <h3>What is Aiken?</h3>
  </summary>
  <p>
    Aiken is a new{" "}
    <strong>
      programming language and toolkit for developing smart contracts
    </strong>{" "}
    on the{" "}
    <a href="https://cardano.org/" target="_blank">
      Cardano
    </a>{" "}
    blockchain. It is geared towards robustness and developer experience.
    <br />
    Aiken takes inspiration from many modern languages such as Gleam, Rust, and
    Elm, which are known for friendly error messages and an overall excellent
    developer experience. We believe Cardano deserves a dedicated language with
    these kinds of features, developed in the open with the community.
  </p>
</details>

  <details>
    <summary><h3>Why build another smart contract platform?</h3></summary>
    <p>
      We love functional programming. Yet, we couldn't help to think that the
      current Haskell framework wasn't <i>quite right</i> in terms of developer
      experience and felt we could do something about it. So we came up with a
      set of shared goals we kept in mind while building Aiken:
    </p>
    <ul>
      <li class="icon-coffee">Writing smart contracts should be <strong>easy and
      safe</strong>. You should be able to get started in minutes, not days,
      and rapidly build confidence that your on-chain code is doing what is
      intended.</li>

      <li class="icon-heart">We want a complete and delightful experience. A modern
      blockchain deserves a modern smart contract toolkit. <strong>You should
      be and feel productive</strong> when writing contracts. This includes
      editor integrations such as LSP and tree-sitter, beautiful error
      messages, rapid and intuitive test feedback loop and easy-to-produce
      documentation.</li>

      <li class="icon-fast-forward">We want there to be <strong>as little configuration as
      possible</strong>. It should work out of the box and establish reasonable
      opinionated conventions with the community.</li>

      <li class="icon-grid">We want to have a <strong>modular design</strong> such
      that components can be picked and chosen as needed. Like the Unix
      philosophy. Aiken is only one part of a much bigger picture;. While we
      firmly believe it is a great tool, we want to encourage other approaches
      and favour interoperability as much as possible. </li>
    </ul>

  </details>

{" "}

<details>
  <summary>
    <h3>I thought Cardano smart contracts had to be written in Haskell?</h3>
  </summary>
  <p>
    This is a common misconception. The current Cardano node implementation does
    indeed happen to be written in Haskell. The virtual machine for executing
    smart contracts that comes baked into the node is also implemented in
    Haskell. But that does not mean that it is Haskell which is executed by the
    smart contract virtual machine.
    <br />
    The virtual machine is a language interpreter which executes a smart
    contract language called <i>'Untyped Plutus Core'</i> (abbrev.
    <code>UPLC</code>) often referred to simply as <i>'Plutus'</i>.
    <br />
    Yet UPLC isn't something developers are expected to write by hand. Instead, it
    is a compilation target (like WebAssembly for the world wide web). Oddly enough,
    until recently, the only established framework that produced UPLC from a high-level
    syntax was called <i>'Plutus-Tx'</i> and happened to be a Haskell framework.
    <br />
    Aiken changes the game by introducing a new framework that compiles straight
    to UPLC.
  </p>
</details>

  <details>
    <summary><h3>Can I write off-chain/backend code with Aiken</h3></summary>
    <p>
      Our main goal is to improve the smart contract development experience
      <strong>for the Cardano blockchain</strong>. By developing Aiken as a
      bespoke language, we can better align its features with what is truly
      needed for on-chain development.
      <br/>
      We want to keep the language simple and manageable. In a decentralized
      architecture, smart contracts present a significant challenge where a
      single flaw can escalate into a multi-million financial exploit. More so,
      on-chain code typically represents a small fraction of an entire DApp
      source code. While writing contracts should be as straightforward as possible,
      smart contracts ought to be optimized for review, audit, and static
      analysis.
      <br/>
      Hence, Aiken is <strong>not</strong> intended as a general-purpose
      language. Rather, it focuses on Cardano and aims for a high-quality
      toolkit for developing reliable smart contracts with confidence.
    </p>
  </details>
