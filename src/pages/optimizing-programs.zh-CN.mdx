## 在优化之前

优化代码有时可能与直觉相悖，尤其是在智能合约的背景下。虚拟机及其相关的成本模型有时可能令人困惑，并以难以预料的方式运作。

因此，在进行任何优化工作之前，建立一些基准测试（benchmarks）至关重要。这些基准测试应涵盖简单和复杂的场景，以便轻松识别变更带来的影响。有时，一项变更可能会引入一次性成本，虽然略微增加简单案例的成本，但能让更复杂的场景显著改善。

### 编写基准测试

编写基准测试的一个好策略是编写一个简单的 _测试_，从预先构建的上下文和赎回者（redeemer）执行一个或多个验证器。这可以让您获得尽可能接近带有“真实交易”之“真实场景”的概览。

例如：

```aiken
use my_validator

test baseline() {
  my_validator.main.withdraw(
    baseline_redeemer,
    baseline_credential,
    baseline_transaction,
  )
}
```

### 使用 `const`

Aiken 的常量在 **编译期间** 会被完整求值，并在使用的位置被 **内联（inlined）**。因此，强烈建议将基准测试的准备代码定义为常量。这不仅可以將测试数据（fixture）代码与基准测试执行分离，还能测量更接近真实执行的成本。

```aiken
const baseline_transaction: Transaction = Transaction {
  ..transaction.placeholder,
  inputs: some_inputs(3),
}

test baseline() {
  my_validator.main.withdraw(
    baseline_redeemer,
    baseline_credential,
    baseline_transaction,
  )
}
```

### 使用 `Fuzzer`

Fuzzers 是编写测试数据的一种非常实用的方式。特别是交易，可以使用 [`fuzz/cardano`](TODO) 中的原语轻松创建。例如：

```aiken
use aiken/fuzz
use cardano/fuzz as cardano

pub fn some_basic_input() -> Fuzzer<Input> {
  let output_reference <- fuzz.and_then(cardano.output_reference())
  let address <- fuzz.and_then(cardano.address())
  fuzz.constant(Input {
    output_reference,
    output: Output {
      address,
      value: min_ada_value,
      datum: NoDatum,
      reference_script: None,
    }
  })
}
```

一个好的经验法则是从标准库的 `transaction.placeholder` 开始，逐步向交易添加元素以使其有效。Fuzzers 对于交易中可以任意设置的元素非常方便，而对于其他元素，您则可以坚持使用已知的值。

```aiken
const transaction = Transaction {
  ..transaction.placeholder,
  inputs: [
    some_input_with_programmable_tokens(
      [(policy_programmable_token, asset_name_programmable_token, 42)],
      "programmable tokens input",
    ),
    some_input_fuel("fuel input"),
  ],
  outputs: [
    some_output_with_programmable_tokens(
      [(policy_programmable_token, asset_name_programmable_token, 28)],
    ),
    some_output_with_programmable_tokens(
      [(policy_programmable_token, asset_name_programmable_token, 14)],
    ),
    some_output_change("change output"),
  ],
}
```

### 标准库：好还是坏？

让我们在深入探讨之前涵盖最后一点：标准库。您应该使用它吗？当然，是的。它会损害您的程序性能吗？在某种程度上，是的。标准库经过了 **相当合理的优化**，但它是针对 **正确性** 和 **易用性** 进行了调整。其主要目标是让您入门并提供方便。

然而，在需要的地方进行局部替换很容易。标准库中的大多数函数都是独立的，易于内联且可以特化。因此，建议始终从标准库开始编写最 _显而易见的正确_ 代码，然后再考虑哪里可以优化。

许多优化实际上是领域特定的，需要内在知识才能真正奏效。在设计智能合约时，关于代码如何编写的优化不应该是首要任务（而应该只是架构层面的关注量）。一旦您的链上代码基本成型，反思您在程序的关键部分对标准库的使用是一件好事：也许您不需要这个特定函数提供的所有泛型，或者您可以使用另一个函数的更简单、更直接的递归实现。

在验证器中，有一些来自标准库的函数是您特别需要留意并避免使用的。这些函数通常只适合测试，而不适合关键路径。这些警告信号是：

- `assets.{flatten, flatten_with, restricted_to}{:ak}`
- `dict.{from_pairs, keys, map, values}{:ak}`
- `list.{count, flat_map, map, reverse, sort, zip}{:ak}`

您几乎肯定永远不想在验证器中使用其中的任何一个。

### 使用简单（更简单）的结构

除非它来自 _外部世界_（即 datum 或 redeemer），否则请避免构建大型记录。Aiken 是一种直接在编码对象（又名 `Data`）上运行的语言。当对象在脚本执行前已经预先构建好时（如 datum、redeemer 或交易本身的情况），这非常方便。

然而，为了在多个交易元素之间传递上下文而构建大型记录通常会带来显著的成本。因此，当您实际上不需要具体化中间结构时，请优先使用带有显式参数的函数。

特别地，如果您能避开它，就不要构建 `Value`，并尽可能优先使用 `Dict` 或 `Pairs` 而非 `Value`。

`Value` 维护了两个重要的不变量：它不包含零数量的资产或资产为空的策略。如果您不依赖这些不变量，您可以安全地降级到 `Dict`。

`Dict` 维护了两个重要的不变量：它们的键按升序排列且不包含重复项。如果您不依赖这些不变量，您可以安全地降级到 `Pairs`。


### 对不可出的搜寻使用快速递归

这是快速失败策略的一个更具体的版本，适用于 _“不可出错的搜寻”_。当在集合中寻找特定元素且没有任何可能的错误恢复时，就会发生这种情况：如果不存在，那么这就是一个错误，整个验证器必须失败。

这种情况在验证器中其实非常常见，特别是在处理作为协议一部分的元素时。

### 优先使用二分搜寻而非线性搜寻

拥有多个条件链是非常常见的，如果以最天真的方式编写，可能会导致不必要的求值。当条件在某种程度上是等概率的（即没有明显的不平衡使得某些条件比其他条件更容易满足）时，重新构建并嵌套某些 if/then/else 以执行二分搜寻可能会很有用。

  </Cards>

  ```aiken
  // 线性搜寻，不理想。
  fn mod32(i) {
    if i < 32 { 0 }
    else if i < 64 { 1 }
    else if i < 96 { 2 }
    else if i < 128 { 3 }
    else if i < 160 { 4 }
    else if i < 192 { 5 }
    else if i < 224 { 6 }
    else { 7 }
  }
  ```
  </Tabs.Tab>

  <Tabs.Tab>
  <Cards>
    <Cards.Card title="mem=37.06K" href="#favor-binary-searches-over-linear-searches" />

  ```aiken
  fn mod32(i) {
    if i < 128 {
      if i < 64 {
        if i < 32 { 0 } else { 1 }
      } else {
        if i < 96 { 2 } else { 3 }
      }
    } else {
      if i < 192 {
        if i < 160 { 4 } else { 5 }
      } else {
        if i < 224 { 6 } else { 7 }
      }
    }
  }
  ```
  </Tabs.Tab>
  <Tabs.Tab>

  ```aiken

      mod32(47) == 1,
      mod32(89) == 2,

      mod32(200) == 6,
      mod32(225) == 7,
    }
  }
  ```
  </Tabs.Tab>

在此示例中，我们根据所选的 0 到 255 之间的某个整数值进行分支。第一种形式逐个评估每个条件，导致 **线性搜寻**，平均评估次数为 `(n + 1) / 2`。因此对于 `n=7`，平均评估次数为 `4` 次。

然而，建议的示例安排了条件以减少每次通过时执行的评估量。它执行 **二分搜寻**，从而产生 `log2(n)` 次评估。因此对于 `n=7`，平均评估次数为 `3` 次。

此外，二分搜寻具有更 **可预测** 的优点。在前面的示例中，它不仅平均进行 3 次条件评估，而且每次通过始终评估 3 個条件。与“不建议”的示例不同，后者有时评估一个条件，有时评估三个，有时评估七个，等...

### 将开销低且可能性高的检查放在前面

当使用 `and{:ak}` 或 `or{:ak}` 链接条件時，顺序很重要。Aiken 会短路布尔运算符，这意味着 `or{:ak}` 的第一个满足的分支避免了求值其他分支，而 `and{:ak}` 的第一個失败的分支会停止其余部分。

所以，在可能的情况下，请优先放置同时具备以下特性的检查：

1. 求值开销更低，且
2. 更有可能决定结果（即更频繁地为 `True{:ak}`）

<Tabs items={["不建议", "建议"]}>
<Tabs.Tab>

```aiken
or {
  input.output.value |> assets.has_nft_strict(my_nft),
  input.output.address.payment_credential != my_script_credential,
}
```
</Tabs.Tab>
<Tabs.Tab>

```aiken
or {
  input.output.address.payment_credential != my_script_credential,
  input.output.value |> assets.has_nft_strict(my_nft),
}
```
</Tabs.Tab>
</Tabs>

在这个示例中，比较凭证是一个直接且可预测的检查。而检查数位以确定是否存在特定 NFT 则更为复杂。由于第一个条件可能已经足以决定整个表达式，将其放在前面可以为运行时提供更多提前停止的机会。

### 优先使用 `Data` 相等性而非手动结构比较

Aiken 程序在编码后的 `Data` 上运行，直接比较 `Data` 值通常出奇地高效。如果预计两个值在结构上匹配，则原始相等性检查几乎总是比手动重新构建该逻辑更便宜。

这在处理表示数值或状态快照的 datum 时特别有用。

标准库已经公开了对此有用的辅助函数。例如，[`assets.match{:ak}`](https://aiken-lang.github.io/stdlib/cardano/assets.html#match) 可以将运行时的 `Value{:ak}` 与 `Data{:ak}` 表示形式进行比较，同时允许您参数化应如何检查 lovelace：

```aiken
pub fn match(
  left: Value,
  right: Data,
  assert_lovelace: fn(Lovelace, Lovelace) -> Bool,
) -> Bool
```

更普遍的教训是，如果一个结构的大部分应保持不变，通常最好直接比较不变的部分，并隔离预计会变化的部分。

例如，一个强大的优化模式是将结构快速拆分为：

* 可变区域之前的部分，
* 可变区域本身，
* 以及可变区域之后的部分。

然后，您可以通过数据相等直接比较稳定区域，并仅详细检查变化的部分。

```aiken
let input_tokens_before, input_tokens_at, input_tokens_after <- split_at(input.value, policy_id)
let output_tokens_before, output_tokens_at, output_tokens_after <- split_at(output.value, policy_id)
```

然后：

```aiken
expect and {
  (input_tokens_before == output_tokens_before)?,
  (input_tokens_after == output_tokens_after)?,
  (input_tokens_at != output_tokens_at)?,
}
```

这通常比在完整的 `Value{:ak}` 结构上重新计算完整的语义比较便宜得多。

### 当返回多个值时使用回传 (Backpassing)

返回大型元组或记录很方便，但也意味着要构建中间值，然后立即再次解构它们。在热点路径中，这种开销可能会变得很明显。

回传允许您将“延续” (continuation) 直接线程化到函数中。

### 如果回传不是一个选项，优先选择 `Pair` 而非 2-tuple

当您需要精确返回两个值且回传会使代码可读性降低時，`Pair<a, b>{:ak}` 通常比 `(a, b){:ak}` 略微更可取。

两者都可以轻松访问：

* `pair.1st{:ak}` / `tuple.1st{:ak}`
* `pair.2nd{:ak}` / `tuple.2nd{:ak}`

但 `Pair{:ak}` 与字典和基于 Pairs 的 API 集成更自然，因此它往往能与标准库的其余部分更好地组合。

这通常不算是改变游戏规则的优化，但在处理键值形状的数据時，这是一個很好的默认设置。

### 避免重复遍歷

多次遍歷同一個集合是积累可避免成本最簡單的方法之一。在验证器中，集合通常不那么大，但重复遍歷仍会迅速增加开销。

如果您需要从同一個列表獲取多個派生值，请尝试在一次遍歷中计算它们。

```aiken
  return: fn(Int, List<Int>) -> result,
) -> result {
```

```aiken
        count + 1,
        if n >= 0 {
          [n, ..positive]
        } else {
          positive
```

### 用查表取代昂贵的计算

一個反复出现的优化主题是计算通常比查表更昂贵。如果函数运行在有边界的定义域上，则可能预先计算所有结果并将它们存储在紧凑的查找结构中。

对于非常小的定义域，位元组数组已经可以充当查找表。确切的实现可能有所不同，但在以下情况下，该技术很有用：

* 输入空间有界,
* 输出可以紧凑编码,
* 且计算足够昂贵值，得进行预计算。

标准库中一個很好的例子是 `math.pow2{:ak}`，它结合了位元组数组查表和展开递归以提高效率：

```aiken

  if e <= 0 {

      1
    } else {

  } else {
```

### 对大型注册表使用 Merkle Patricia Forestry

对于更大的查表空间，简单的位元组数组不再足够。这就是 [Merkle Patricia Forestry](https://github.com/aiken-lang/merkle-patricia-forestry) 可以发揮作用的地方：它使您能够以链上非常便宜的成员资格检查来表示任意大的键值注册表。

因此，与其：

```aiken
let y = expensive_function(x)
```

您有时可以这样做：

```aiken
expect mpf.member(y, root)
```

其中 `root{:ak}` 是在链下提交的预先计算的、经验证的结构。

一般模式是：

1. 在链外计算一个大型注册表,
2. 在链上提交其根,
3. 通过 redeemer 传递证明或查询的值,
4. 验证成员资格而不是重新计算。

对于定义域有界但计算昂贵（> 总执行预算的 1%）的确定性函数，这特别有吸引力。

这带給我们一個更广泛的原则：验证器并不总是需要从头开始进行计算。通常，验证交易提供的值是否正确就足够了。

这是链上编程中 **最重要的优化模式之一**：只要正确性仍能在链上以便宜的成本进行验证，就将工作移至链下。

### 利用账本不变量

最后一種优化技术并非真正关于代码形状，而是关于对账本有足够的了解，以便依靠其保证。

验证器检查的許多结构已经满足强大的不变量。例如：

* 输入按字母顺序排序,
* 价值按策略和资产名排序,
* redeemer 和 datum 通过哈希索引,
* 输出价值绝不包含负数量,
* 输出价值始终包含 ADA。
* 铸造价值绝不包含 ADA。
* 等等...

这些不仅仅是方便的事实；它们通常是编写便宜得多的算法的关键。

如果您知道结构已经排序，您可以线性地合并或比较它，而不是重新排序。如果您知道数量不能为负，则可以避免防御性的规范化逻辑。如果您知道映射没有重复键，您可以使用更直接的遍歷策略。

最有效的优化通常不是「聪明的代码技巧」，而是与账本已经保证的不变量紧密对齐的代码。

## 结语

Aiken 中的大多数优化工作都遵循一些反复出现的主题：

* 尽早失败而不是恢复,
* 避免不必要地重建丰富的结构,
* 尽可能少地遍歷集合,
* 在实际時用查表替换计算,
* 只要有可能就用验证替换计算,
* 并依靠账本已经赋予您的不变量。

良好的工作流程通常是：

1. 编写最简单的显然正确的验证器,
2. 对其进行端到端基准测试,
3. 识别实际的热路径,
4. 只优化那些部分,
5. 在每次重大更改后重新基准测试。
